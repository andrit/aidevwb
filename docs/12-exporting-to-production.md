# Exporting Workbench Infrastructure to Production

## The Problem

You've built a RAG application, a chatbot, or an agent using the workbench. It works. Now you need to ship it. Your project needs its own Postgres with pgvector, its own Redis, its own API layer — independent of the workbench. Today you'd manually recreate all of this. That's error-prone and slow.

The workbench should have an `export` command that generates a self-contained production stack for your project, derived from the workbench infrastructure but owned entirely by the project.

## What Gets Exported

Not everything in the workbench is relevant to your shipped project. The export is selective based on what your project actually uses:

```
Workbench Component      → Exported As                        → When
────────────────────     ─────────────                        ──────
supabase-db (Postgres)   → docker-compose service + migrations → Always if project uses RAG
                         → OR Terraform module for RDS         → If deploying to cloud
pgvector + hybrid_search → SQL migration files                 → If project uses RAG
Redis + BullMQ           → docker-compose service              → If project uses async jobs
                         → OR Terraform module for ElastiCache → If deploying to cloud
MCP server (Fastify)     → standalone API service              → If project exposes RAG via API
RAG worker (Python)      → standalone worker service           → If project ingests documents
OTel + Grafana           → docker-compose services             → If project wants observability
```

Components that are **never** exported:
- `claude-code` container (development tool only)
- MCP bridge (Claude Code integration only)
- Slash commands (development workflow only)
- `.workbench/` directory (development state only)
- Supabase Kong/REST gateway (replaced by the project's own API)

## What the Export Produces

```bash
make export-stack NAME=nexus DIR=~/code/nexus
```

This generates a `stack/` directory inside your project. The export follows the same safety principle as project import: **never overwrite existing files.** If `stack/` already exists (from a previous export or because the project uses that directory name), the export warns and asks before proceeding. All generated files go inside `stack/` — the export never writes to the project root or modifies any existing project file.

```
~/code/nexus/
├── src/                          ← your existing code (UNTOUCHED)
├── stack/                        ← GENERATED by export (new directory)
│   ├── docker-compose.yml        ← production services (Postgres, Redis, API, worker)
│   ├── docker-compose.dev.yml    ← dev overrides (ports exposed, debug logging)
│   ├── .env.example              ← required env vars for the stack
│   ├── Dockerfile.api            ← your project's API server (derived from mcp-server)
│   ├── Dockerfile.worker         ← your project's worker (derived from rag-worker)
│   ├── migrations/
│   │   ├── 001_extensions.sql    ← pgvector + pg_trgm
│   │   ├── 002_documents.sql     ← documents table
│   │   ├── 003_chunks.sql        ← chunks table with vector column
│   │   └── 004_hybrid_search.sql ← hybrid search function
│   ├── terraform/                ← cloud deployment (optional)
│   │   ├── main.tf               ← RDS + ElastiCache + ECS
│   │   └── variables.tf
│   └── README.md                 ← how to run the exported stack
└── ...
```

## Export Modes

### Mode 1: Docker Compose Stack (default)

For projects that deploy via Docker Compose, Docker Swarm, or any container orchestrator:

```bash
make export-stack NAME=nexus FORMAT=compose
```

Produces a `docker-compose.yml` with:
- Postgres 15 + pgvector (no Supabase overhead — just raw Postgres)
- Redis 7 (if the project uses queuing)
- Your API service (Fastify, derived from mcp-server but stripped of MCP/workbench-specific code)
- Your worker service (Python, derived from rag-worker)

The exported compose file is **standalone** — it has no references to the workbench. You can copy it to any machine, run `docker compose up`, and it works.

### Mode 2: Terraform Modules

For projects deploying to AWS (or adaptable to GCP/Azure):

```bash
make export-stack NAME=nexus FORMAT=terraform
```

Produces Terraform modules derived from the workbench's `infra/terraform/modules/` but scoped to only what the project needs. No workbench-specific tagging, no multi-project registry — just your project's infrastructure.

### Mode 3: Kubernetes Manifests

```bash
make export-stack NAME=nexus FORMAT=k8s
```

Produces Kubernetes Deployment, Service, ConfigMap, Secret, and PersistentVolumeClaim manifests. Useful for projects deploying to EKS, GKE, or self-hosted K8s.

### Mode 4: Bare Migrations Only

For projects that already have their own infrastructure and just need the database schema:

```bash
make export-stack NAME=nexus FORMAT=migrations-only
```

Produces only the SQL migration files. Your project runs them against its own Postgres.

## What Changes in the Exported Code

The exported API service is derived from the workbench's MCP server but with workbench-specific code removed:

**Removed:**
- MCP stdio bridge and tool definitions (your app doesn't need Claude Code integration)
- Slash command handling
- Workbench-specific health check responses
- Multi-project routing

**Kept:**
- Fastify HTTP routes (ingest, query, status)
- Zod schemas for validation
- Hybrid search service
- Embedding service (OpenRouter)
- LLM service (Claude)
- BullMQ queue integration
- All configuration via environment variables

**Customizable:**
- System prompts (change from generic knowledgebase assistant to your app's persona)
- Search weights (tune vector_weight and text_weight for your domain)
- Chunk size and overlap
- API route paths and authentication

The exported code is meant to be **forked and modified**. It's a starting point, not a dependency. Once exported, you own it completely.

## Data Export

The export also handles migrating your development data to the production database:

```bash
# Export the project's knowledgebase data
make export-data NAME=nexus

# Produces: stack/seed-data.sql.gz
# Contains: documents + chunks + embeddings from the workbench's dev database
```

This lets you seed your production database with the same knowledgebase you developed against. In production, you'd then set up your own ingestion pipeline (or use the exported worker service) for ongoing document updates.

## The Export Workflow

```
1. Develop in the workbench (build, ingest, query, iterate)
2. When ready to ship:
     make export-stack NAME=nexus FORMAT=compose
3. Review and customize the generated stack/
4. Move stack/ into your project's deployment infrastructure
5. Seed the production database:
     cd stack && docker compose up -d
     gunzip -c seed-data.sql.gz | docker exec -i postgres psql -U postgres -d nexus
6. Your project is now independent. The workbench is no longer involved.
```

## What This Means for the "Ship" Pipeline Stage

The pipeline from doc 11 gains a concrete SHIP step:

```
INIT → CONTEXT → BUILD → TEST → OBSERVE → SHIP → MAINTAIN
                                             │
                                   make export-stack
                                   make export-data
                                   customize stack/
                                   deploy independently
```

After SHIP, the project is fully independent. The workbench can continue to be used for MAINTAIN (re-ingesting docs, tuning search, debugging with traces), but the production system doesn't need or reference the workbench.
