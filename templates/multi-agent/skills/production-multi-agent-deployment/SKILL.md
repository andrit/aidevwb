# Production Multi-Agent Deployment

## When to use

When a multi-agent team is ready to run without the workbench. The workbench provides Redis, the MCP API, and debug holds during development — in production, the team needs its own Redis instance and must point to the project's exported API. Activate when deploying a team, when bus messages are accumulating without bound, or when the user asks "how do I ship the agent team?"

## Prerequisites

- Team working and eval-passing in the workbench (`run-multi-agent-eval` skill complete)
- Each agent individually passing the `production-agent-deployment` checklist
- `make export-stack NAME=<project> FORMAT=compose` run — generates standalone compose

## Step 1 — Configure Standalone Redis

The workbench's Redis is shared across all projects. In production, the team gets its own Redis instance.

```yaml
# docker-compose.prod.yml — generated by make export-stack, review and adjust
services:
  redis:
    image: redis:7-alpine
    command: >
      redis-server
      --maxmemory 512mb
      --maxmemory-policy allkeys-lru
      --appendonly yes
      --appendfilename "agent-bus.aof"
    volumes:
      - redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
    restart: unless-stopped

  api:
    image: myproject-api:latest
    environment:
      REDIS_URL: redis://redis:6379
      DATABASE_URL: ${DATABASE_URL}
      # ... other env vars

  orchestrator:
    image: myproject-orchestrator:latest
    environment:
      WORKBENCH_API: http://api:3000       # ← points to OWN api, not workbench
      WORKBENCH_PROJECT: ${PROJECT_NAME}
      REDIS_URL: redis://redis:6379
    depends_on:
      redis:    { condition: service_healthy }
      api:      { condition: service_healthy }

    image: myproject-researcher:latest

      redis:    { condition: service_healthy }

volumes:
  redis-data:
```

## Step 2 — Set Bus Message TTL

Development bus channels accumulate messages indefinitely. In production, apply a TTL so Redis doesn't grow unbounded.

```python
# agents/lib/bus.py — bus wrapper with TTL
import redis.asyncio as aioredis
import json
import os

REDIS_URL = os.environ["REDIS_URL"]
BUS_MAXLEN = int(os.environ.get("BUS_MAXLEN", "10000"))  # keep last 10k messages per channel
BUS_TTL_SECONDS = int(os.environ.get("BUS_TTL_SECONDS", str(7 * 24 * 3600)))  # 7 days

_redis: aioredis.Redis | None = None

async def get_redis() -> aioredis.Redis:
    global _redis
    if _redis is None:
        _redis = await aioredis.from_url(REDIS_URL)
    return _redis

async def publish(channel: str, payload: dict) -> None:
    r = await get_redis()
    message = json.dumps(payload)
    # XADD with MAXLEN trims the stream to the last BUS_MAXLEN messages
    await r.xadd(channel, {"data": message}, maxlen=BUS_MAXLEN, approximate=True)
    # Set TTL on the stream key itself
    await r.expire(channel, BUS_TTL_SECONDS)

async def read(channel: str, group: str, count: int = 10) -> list[dict]:
    r = await get_redis()
    # Create consumer group if it doesn't exist
    try:
        await r.xgroup_create(channel, group, id="0", mkstream=True)
    except Exception:
        pass  # BUSYGROUP — already exists

    messages = await r.xreadgroup(
        group, "worker", streams={channel: ">"}, count=count, block=1000
    )
    results = []
    for _, entries in (messages or []):
        for msg_id, fields in entries:
            data = json.loads(fields[b"data"])
            results.append({"id": msg_id.decode(), **data})
            await r.xack(channel, group, msg_id)
    return results
```

## Step 3 — Persist Team Run Results

In development, results are printed or logged. In production, persist every team run for audit and retry.

```sql
-- supabase/migrations/015_team_runs.sql
CREATE TABLE team_runs (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  team_name    TEXT NOT NULL,
  task         TEXT NOT NULL,
  status       TEXT NOT NULL DEFAULT 'running',  -- running, completed, partial, failed
  result       JSONB,
  agents_ran   TEXT[] DEFAULT '{}',
  agents_failed TEXT[] DEFAULT '{}',
  total_cost_usd NUMERIC(10,6) DEFAULT 0,
  total_turns  INT DEFAULT 0,
  started_at   TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ,
  error_message TEXT
);

CREATE INDEX ON team_runs(team_name, status);
CREATE INDEX ON team_runs(started_at DESC);
```

```python
# agents/orchestrator.py — persist run results
from agents.lib.db import get_db
import uuid

async def run_team(task: str) -> dict:
    run_id = str(uuid.uuid4())
    db = await get_db()

    await db.execute(
        "INSERT INTO team_runs (id, team_name, task) VALUES ($1, $2, $3)",
        run_id, "research-team", task
    )

    try:
        result = await execute_team(task, run_id)

        await db.execute(
            """UPDATE team_runs
               SET status='completed', result=$2, agents_ran=$3,
                   total_cost_usd=$4, total_turns=$5, completed_at=NOW()
               WHERE id=$1""",
            run_id, result["output"], result["agents"],
            result["total_cost"], result["total_turns"]
        )
        return result

    except Exception as e:
        await db.execute(
            "UPDATE team_runs SET status='failed', error_message=$2, completed_at=NOW() WHERE id=$1",
            run_id, str(e)
        )
        raise
```

## Step 4 — Per-Team Cost Envelope

Apply a cost budget at the team level, not just per agent.

```python
# agents/lib/team_budget.py
from dataclasses import dataclass, field

@dataclass
class TeamBudget:
    run_id:              str
    max_cost_usd:        float = 5.0     # halt team if total exceeds this
    max_total_turns:     int   = 100     # across all agents
    agent_costs:         dict  = field(default_factory=dict)

    @property
    def total_cost_usd(self) -> float:
        return sum(self.agent_costs.values())

    @property
    def total_turns(self) -> int:
        return sum(self.agent_costs.values())  # extend to track turns per agent

    def record_agent_cost(self, agent_name: str, cost: float) -> None:
        self.agent_costs[agent_name] = self.agent_costs.get(agent_name, 0) + cost
        if self.total_cost_usd > self.max_cost_usd:
            raise TeamBudgetExceededError(
                f"Team budget ${self.max_cost_usd:.2f} exceeded: ${self.total_cost_usd:.4f}"
            )

class TeamBudgetExceededError(Exception):
    pass
```

## Step 5 — Smoke Test the Deployed Team

```bash
#!/usr/bin/env bash
# scripts/team-smoke-test.sh
set -e

API="${ORCHESTRATOR_URL:-http://localhost:8080}"

echo "=== Multi-Agent Team Smoke Test ==="

# 1. All agents live
for agent in orchestrator researcher writer; do
  status=$(curl -sf "http://$agent:8080/health/live" | jq -r '.status' 2>/dev/null || echo "fail")
  [ "$status" = "ok" ] && echo "✓ $agent liveness" || { echo "✗ $agent not live"; exit 1; }
done

# 2. Redis reachable from orchestrator
redis_check=$(docker exec orchestrator redis-cli -u "$REDIS_URL" ping 2>/dev/null || echo "FAIL")
[ "$redis_check" = "PONG" ] && echo "✓ Redis" || { echo "✗ Redis not reachable: $redis_check"; exit 1; }

# 3. Bus publish/read round trip
curl -sf -X POST "$API/bus/test" > /dev/null 2>&1 && echo "✓ Bus round-trip" || echo "⚠ Bus test skipped"

# 4. Minimal team run
result=$(curl -sf -X POST "$API/run" \
  -H "Content-Type: application/json" \
  -d '{"task":"Summarize: The sky is blue.","max_cost_usd":0.10}')
echo "$result" | jq -r '.status' | grep -q "completed" \
  && echo "✓ Team run completed" \
  || { echo "✗ Team run failed: $result"; exit 1; }

echo "=== All smoke tests passed ==="
```

## Checklist

- [ ] Standalone Redis in `docker-compose.prod.yml` — not shared with workbench
- [ ] `BUS_MAXLEN` and `BUS_TTL_SECONDS` set in env — channels don't grow unbounded
- [ ] All agent `WORKBENCH_API` env vars point to project's own API, not `mcp-server:3100`
- [ ] `team_runs` table persists every run — status, result, cost, agents involved
- [ ] `TeamBudget` wraps orchestrator — team halts if total cost exceeds envelope
- [ ] Each agent individually passes `production-agent-deployment` checklist
- [ ] Smoke test passes against staging before production deploy
- [ ] Redis `maxmemory` policy set — evicts old keys under memory pressure

## Common mistakes

**Shared Redis between workbench and production** — if the production team writes to the workbench's Redis, messages from development runs appear in the production bus and vice versa. Always give production its own Redis instance.

**No bus message TTL** — Redis Streams grow indefinitely without `MAXLEN` or TTL. On a busy team, a week of messages can consume gigabytes. Set `BUS_MAXLEN` from day one.

**All agents sharing one `WORKBENCH_API` default** — if the default URL is wrong and one agent silently falls back to it, that agent's memory calls fail silently. Remove all defaults; require the variable to be set; test readiness endpoints that check the API connection.
